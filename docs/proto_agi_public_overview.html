Public-Facing Architecture Overview

Proto-AGI Stack — Persistent, Goal-Driven Intelligence Architecture

This document outlines a practical path from a capable local assistant stack to a proto-AGI architecture: a persistent system that can observe, remember, plan, act, evaluate, and improve across time. The goal is not to make inflated claims, but to define a real engineering stack for building a stronger, more autonomous intelligence system from existing components such as local controllers, ARC-style memory/receipts, deterministic runtimes, and simulation-backed world modeling.

Core Position
A system, not a single model
Proto-AGI is not just a chatbot or router. It is an integrated stack that combines memory, reasoning, goal management, planning, tool use, feedback, and persistence into one continuous loop.
Practical Goal
From assistant → agent
The target is a system that can take a high-level goal, decompose it, execute across multiple steps, detect failures, revise its plan, and improve over time without full manual steering.
The Complete Proto-AGI Loop
observe → update world model → choose goal → plan → act → evaluate → learn → repeat
Architecture Stack
1. Perception Layer
Handles incoming text, code, structured data, files, and later multimodal signals. Converts raw inputs into usable state updates.
2. World Model
Maintains structured understanding of projects, tools, environment state, active branches, known facts, and causal relationships.
3. ARC Memory + Receipts
Stores episodic history, semantic facts, procedural know-how, and strategic patterns. Every action emits provenance receipts.
4. Goal Engine
Creates, prioritizes, decomposes, and tracks goals. Supports user-provided goals and system-generated subgoals.
5. Planner
Expands goals into action graphs, estimates cost and likelihood, branches when needed, and revises plans when outcomes change.
6. Action / Tool Layer
Executes shell commands, scripts, build pipelines, file operations, API calls, and future wrapped tools through controlled interfaces.
7. Evaluation Layer
Measures whether actions helped the current goal, detects wasted work, classifies failures, and captures what should be remembered.
8. Learning Layer
Promotes repeated success patterns into reusable procedures, adjusts strategy weights, and improves future planning and tool selection.
9. Persistent Agent Loop
The always-running process that keeps the whole stack alive across time instead of resetting on every prompt.
What Existing Local Assistant Stacks Already Cover
  • Prompt handling
  • Command routing
  • Model/tool selection
  • Fallback layers
  • Basic script/code generation
What Must Be Added For Proto-AGI
  • Persistent goal state
  • World-model updates
  • Autonomous planning across steps
  • Structured memory beyond logs
  • Closed-loop evaluation and learning
What Should Wait Until Later
  • Consciousness claims
  • Universal autonomy
  • Full world simulation as the core engine
  • Self-rewriting everything
  • Replacing all human oversight
Operational Memory Model
Memory Type Purpose Examples
Episodic What happened Task failed, branch succeeded, build passed, export broke
Semantic What is true Repo layout, tool capabilities, environment facts, API schemas
Procedural How to do something Build sequence, deployment procedure, fix pattern for repeated error
Strategic What tends to work Which planner works best, which tool path is reliable, which branching mode wastes time

Suggested Build Order
Phase What To Build Why It Matters
1 Persistent loop daemon Turns the system from prompt-response into continuous intelligence
2 Goal engine + goal table Creates real agency structure instead of pure user-driven reactions
3 Planner module Converts goals into structured action graphs
4 Tool receipt wrappers Makes every action traceable, evaluable, and reusable
5 Evaluation / learning layer Allows the system to improve instead of repeating failures forever
6 World-model integration Supports stronger planning and causal reasoning across time
Where Simulation Fits
Deterministic runtimes, seed-based systems, and simulation environments are valuable as world-model and testing modules, but they should not be the first definition of the agent itself. They are best used as sandboxes for policy testing, scenario evaluation, and controlled environment learning.
Where Turbo Execution Fits
Task normalization, branch-aware execution, ARC receipts, and resolved-output jumping make the system dramatically more efficient. They do not replace intelligence, but they do create a stronger substrate for a persistent agent loop.
Bottom Line
Proto-AGI starts when the system can pursue goals across time, not just answer prompts.
A serious proto-AGI stack is a persistent, memory-backed, goal-driven, tool-using, self-evaluating system. The practical next step is to connect an assistant/controller layer, ARC-style memory/receipts, a planner, and a continuous loop into one operating intelligence.
